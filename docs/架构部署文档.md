### 1.2 部署说明

| 平台 | 用途 | 说明 |
|------|------|------|
| **Vercel** | 前端部署 | Vue 3 静态资源托管，全球 CDN 加速，自动 HTTPS，SPA rewrite |
| **Railway** | 后端部署 | FastAPI Python 服务，Docker 部署，支持流式响应 SSE |

## 3. 环境配置

### 3.1 环境变量清单

#### 前端环境变量

| 变量名 | 说明 | 开发环境 | 生产环境 |
|--------|------|----------|----------|
| `VITE_API_BASE` | 后端 API 地址 | `http://localhost:8000` | `https://your-backend.railway.app` |

#### 后端环境变量

| 变量名 | 说明 | 必填 | 开发环境 | 生产环境 |
|--------|------|------|----------|----------|
| `SUPABASE_URL` | Supabase 项目地址 | 是 | `https://xxx.supabase.co` | `https://xxx.supabase.co` |
| `SUPABASE_KEY` | Supabase Service Role 密钥 | 是 | `eyJxxx` | `eyJxxx` |
| `SUPABASE_JWT_SECRET` | JWT 密钥（用于本地验证） | 否 | - | `your-jwt-secret` |
| `API_KEY_SECRET` | API Key 加密密钥 | 否* | `local-dev-secret` | `strong-production-secret` |
| `OPENAI_API_KEY` | 默认 OpenAI API Key | 否 | - | `sk-xxx` |
| `CORS_ORIGINS` | 允许的跨域来源 | 否 | `["http://localhost:5173"]` | `["https://your-frontend.vercel.app"]` |

> *`API_KEY_SECRET` 未配置时回退到 `SUPABASE_KEY`，这违反密钥隔离原则。**强烈建议生产环境强制配置独立的加密密钥。**

### 3.2 配置文件示例

#### 前端 `.env`（生产）
```env
VITE_API_BASE=https://your-backend.railway.app
```

#### 后端 `.env`
```env
SUPABASE_URL=https://xxxx.supabase.co
SUPABASE_KEY=eyJxxx（service_role 密钥）
SUPABASE_JWT_SECRET=your-jwt-secret
API_KEY_SECRET=your-strong-secret
OPENAI_API_KEY=
CORS_ORIGINS=["https://your-frontend.vercel.app"]
```

### 4.2 Vercel 部署

#### 1. 安装 Vercel CLI
```bash
npm install -g vercel
```

#### 2. 部署命令
```bash
vercel --prod
```

#### 3. vercel.json 配置
```json
{
  "rewrites": [
    { "source": "/((?!api/).*)", "destination": "/index.html" }
  ]
}
```

> 此配置实现 Vue Router history 模式的 SPA 路由支持：将所有不以 `api/` 开头的路径重写到 `index.html`。

### 4.3 Vite 配置

当前 `vite.config.ts` 配置非常精简，仅注册了 Vue 插件：
```typescript
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'

export default defineConfig({
  plugins: [vue()],
})
```

uvicorn app.main:app --host 0.0.0.0 --port 8000

### 5.3 Railway 部署

#### 1. 连接 GitHub 仓库

在 Railway 控制台：
- New Project → Deploy from GitHub
- 选择仓库和分支

#### 2. 配置环境变量

在 Railway 控制台设置：
- `SUPABASE_URL`
- `SUPABASE_KEY`
- `SUPABASE_JWT_SECRET`
- `API_KEY_SECRET`（**必须配置，不要留空**）
- `CORS_ORIGINS`

#### 3. 部署配置 (railway.json)
```json
{
  "$schema": "https://railway.app/railway.schema.json",
  "build": {
    "builder": "DOCKERFILE",
    "dockerfilePath": "Dockerfile"
  },
  "deploy": {
    "numReplicas": 1,
    "restartPolicyType": "ON_FAILURE",
    "restartPolicyMaxRetries": 10
  }
}
```

### 5.4 Docker 部署

#### Dockerfile
```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

## 6. 数据库配置 (Supabase)

### 6.1 创建项目

1. 访问 [supabase.com](https://supabase.com)
2. 创建新项目
3. 记录项目 URL 和 `anon` / `service_role` 密钥

### 6.2 数据库迁移

在 Supabase SQL Editor 执行以下脚本：

完整建表脚本见 `supabase_schema.sql`（可重复执行），在 Supabase Dashboard → SQL Editor 中运行。

主要特点：
- 包含 `DROP TABLE IF EXISTS ... CASCADE`，可重复执行
- 所有表包含审计字段：`created_by`、`created_at`、`updated_by`、`updated_at`
- 触发器自动填充审计字段（INSERT 设置创建人，UPDATE 刷新更新人/更新时间）
- 行级安全策略（RLS）确保数据隔离

### 6.3 RLS 注意事项

虽然数据库定义了 RLS 策略，但后端使用 `service_role` key 访问 Supabase，**该 key 绕过所有 RLS 规则**。数据隔离完全依赖应用层代码中的所有权校验（`deps.py` 中的 `check_session_owned_cached` 等函数）。

建议：
- 定期审计应用层的所有权校验逻辑
- 考虑对读操作使用用户 JWT 认证（受益于 RLS）

## 12. 常见问题

### 12.1 CORS 错误

**问题**：跨域请求被阻止

**解决**：
1. 检查后端 `CORS_ORIGINS` 配置
2. 确保包含前端域名
3. 重启后端服务

### 12.2 Supabase 连接失败

**问题**：无法连接到 Supabase

**解决**：
1. 检查 `SUPABASE_URL` 和 `SUPABASE_KEY`
2. 确认密钥为 Service Role 密钥
3. 检查 Supabase 项目状态

### 12.3 构建失败

**问题**：`npm run build` 失败

**解决**：
1. 检查 Node.js 版本（推荐 18+）
2. 清除缓存：`rm -rf node_modules && npm install`
3. 检查环境变量配置

### 12.4 流式响应不工作

**问题**：消息不是逐字显示

**解决**：
1. 检查 Nginx 配置是否支持 SSE
2. 确保代理服务器未禁用 chunked transfer
3. 检查前端流式处理代码

### 12.5 JWKS 验证失败

**问题**：JWT 验证总是走远程回退

**解决**：
1. 检查 `SUPABASE_JWT_SECRET` 配置
2. 查看启动日志中 JWKS 预热是否成功
3. 重启后端服务刷新 JWKS 缓存

### 12.6 API Key 解密失败

**问题**：用户配置的 API Key 无法解密

**解决**：
1. 确认 `API_KEY_SECRET` 未变更（密钥变更会导致旧数据无法解密）
2. 如需密钥轮换，需实现多密钥尝试解密机制
3. 当前解密失败会回退明文（安全风险），建议移除此逻辑后重新加密存储
