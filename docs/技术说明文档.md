### 2.4 API 客户端

```typescript
// services/apiClient.ts
const API_BASE = import.meta.env.VITE_API_BASE || ''
```

前端只需配置 `VITE_API_BASE` 一个环境变量，指向后端 API 地址（开发环境为 `http://localhost:8000`）。前端不直接访问 Supabase，所有请求都通过后端代理转发。

`apiClient.ts` 封装了 `get`/`post`/`put`/`delete` 四个方法，基于 `fetch` API，自动添加 `Authorization: Bearer <token>` 请求头。当前实现未配置请求超时（AbortController）。

### 2.5 Chat Service

后端 API 路径如下：

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/api/sessions` | 获取会话列表 |
| POST | `/api/sessions` | 创建新会话 |
| PUT | `/api/sessions/{id}` | 更新会话标题 |
| DELETE | `/api/sessions/{id}` | 删除会话 |
| GET | `/api/sessions/{id}/messages` | 获取会话消息 |

| POST | `/api/chat/stream` | 流式聊天（SSE） |

`chatService` 和 `authService` 均有 localStorage 降级回退机制：后端不可用时自动降级到本地存储。

### 2.6 Markdown 渲染

使用 `marked` + `marked-highlight` + `highlight.js` 实现，在 `MessageBubble.vue` 中通过 `v-html` 渲染。

当前实现：
- `marked.parse()` 渲染 Markdown
- `marked-highlight` 集成 highlight.js 代码高亮
- 自定义 Mermaid 代码块渲染器（生成 `<div class="mermaid">` 标签）
- 流式输出结束后调用 `mermaid.run()` 渲染图表

> **安全注意**：当前 v-html 输出未做 HTML 净化，建议引入 DOMPurify。

### 2.8 认证服务

`authService.ts` 实现了两种模式：
1. **后端模式**：通过 `/api/auth/register` 和 `/api/auth/login` 调用后端 Supabase Auth
2. **降级模式**：后端不可用时使用 localStorage 存储，SHA-256 哈希密码

> **安全注意**：降级模式包含硬编码默认凭据 `admin@admin.com:admin`，仅用于离线开发，不应在生产环境启用。

### 3.4 API Key 加密机制 (security.py)

```python
# 加密：Fernet(sha256(secret)) → encrypt(api_key)
# 解密：Fernet(sha256(secret)) → decrypt(encrypted_key)

# 加密密钥来源：
# 1. 优先使用 API_KEY_SECRET 环境变量
# 2. 未配置时回退到 SUPABASE_KEY（安全风险：密钥隔离原则）
```

> **安全注意**：
> - 解密失败时当前实现回退将密文作为明文返回，这使得加密可被绕过
> - 建议移除回退逻辑，解密失败应抛出异常
> - 建议强制配置独立的 `API_KEY_SECRET`

### 3.5 AI Provider 架构

```
base.py — AIProvider 抽象基类
  ├── chat(messages, model, ...) → AsyncGenerator[str, None]
  └── validate_api_key(api_key) → bool

openai.py — OpenAI 实现
  └── 通过 httpx.AsyncClient 请求 OpenAI API SSE 流
  └── 解析 choices[*].delta.content

deepseek.py — DeepSeek 实现
  └── 兼容 OpenAI API 格式
  └── 通过 httpx.AsyncClient 请求 DeepSeek API SSE 流

factory.py — 工厂方法
  ├── get_provider(name: str) → AIProvider
  └── get_provider_for_model(model: str) → str
      └── "deepseek-*" → deepseek, 其他 → openai
```

### 3.6 认证流程 (deps.py)

1. 从 `Authorization: Bearer <token>` 提取 JWT
2. 尝试本地 JWKS 验证（启动时预热公钥缓存）
3. 本地验证失败时回退到远程 Supabase Auth API
4. 验证通过后返回用户信息

缓存机制：
- `_settings_cache`：用户设置缓存（TTL 5 分钟），包含解密后的 API Key
- `_session_ownership_cache`：会话所有权校验缓存（TTL 5 分钟）

## 4. 数据库设计 (Supabase)

### 4.1 表结构概览

所有表均包含统一审计字段：
- `created_by` (UUID → auth.users) — 创建人
- `created_at` (TIMESTAMPTZ) — 创建时间
- `updated_by` (UUID → auth.users) — 更新人
- `updated_at` (TIMESTAMPTZ) — 更新时间

通过触发器自动填充：INSERT 时 `set_created_by()` 设置 `created_by`/`updated_by`，UPDATE 时 `set_updated_by()` 刷新 `updated_by`/`updated_at`。

### 4.2 表清单

#### settings — 用户配置
| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID PK | 主键 |
| user_id | UUID FK → auth.users | 用户（唯一约束） |
| api_key | TEXT | 用户 API Key（Fernet 加密存储） |
| model | TEXT | 模型（默认 gpt-4o） |
| provider | TEXT | 提供商（默认 openai） |
| temperature | FLOAT | 温度（默认 0.7） |
| max_tokens | INT | 最大 Token（默认 2000） |
| theme | TEXT | 主题（默认 light） |

RLS：`auth.uid() = user_id`

#### sessions — 会话
| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID PK | 主键 |
| user_id | UUID FK → auth.users | 用户 |
| title | TEXT | 标题（默认"新对话"） |

RLS：`auth.uid() = user_id`

#### messages — 消息
| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID PK | 主键 |
| session_id | UUID FK → sessions | 所属会话（级联删除） |
| role | TEXT | 'user' 或 'assistant' |
| content | TEXT | 消息内容 |
| model | TEXT | AI 模型（仅 assistant 有） |

RLS：通过子查询验证 `sessions.user_id = auth.uid()`

## 8. 注意事项

### 8.1 API Key 安全

- 用户配置的 API Key 使用 Fernet 加密存储
- 后端解密后直接用于 AI API 调用，GET /settings 不返回给前端
- **已知风险**：api_key_secret 回退到 supabase_key，解密失败回退明文
- **建议**：强制配置独立加密密钥，移除解密回退逻辑

### 8.2 流式响应兼容

- 使用 Server-Sent Events (SSE) 实现流式响应
- 前端使用 `fetch` + `ReadableStream` 解析
- 确保代理服务器（如 Nginx）支持 SSE

### 8.3 错误处理

- 前端对所有 API 调用进行错误捕获
- 后端使用 Pydantic 进行请求验证
- 错误响应格式统一：`{ "error": "错误信息" }` 或 `{ "detail": "错误信息" }`
- **已知风险**：全局异常处理器返回 `str(exc)`，可能泄露内部信息

### 8.4 架构说明

- 前端**不直接访问 Supabase**，所有数据通过 FastAPI 后端中转，保障 Supabase `service_role` 密钥安全
- 前端 API 调用有 localStorage 回退机制：后端不可用时自动降级到本地存储，便于独立开发 UI
- 认证方式：后端通过 Supabase Auth SDK 处理注册/登录，返回 JWT token 给前端
- 后端使用 service_role key 访问 Supabase，绕过了 RLS 策略，数据隔离完全依赖应用层

### 8.5 已知非功能性问题

详见《架构设计书》第 9 章"已知非功能性问题清单"。
